## 1. 项目概述

**codex-claude-orchestrator** 是一个本地化的多 Agent 编排系统，用于管理和协调多个 Claude Code 工作实例。系统以 tmux 会话为运行容器，通过 MCP Server 暴露编排能力，支持 LLM 驱动的战略决策。

### 核心定位

- **MCP Server 驱动**: MCP Server 内部运行完整监督循环，通过 MCP `sampling/createMessage` 请求 supervisor（LLM）做战略决策
- **可替换 Supervisor**: supervisor 可以是 Codex、Claude API、任意 MCP client、甚至人工 operator
- **本地运行**: 所有组件在本地机器上运行，不依赖云服务
- **契约驱动**: Worker 通过 `WorkerContract` 定义能力、权限、工作空间策略和验收标准
- **事件溯源 (V4)**: 最新版本采用事件溯源架构，所有状态变更为不可变事件

### 入口点

```
orchestrator = "codex_claude_orchestrator.cli:main"
```

定义在 `pyproject.toml` 中，通过 `orchestrator` 命令行工具调用。

## 2. 核心功能

### 2.1 多 Agent 编排

系统可以同时管理多个 Claude Code 工作实例，每个实例运行在独立的 tmux 窗格中。支持三种预设角色组合：

| 模式 | 角色组成 | 适用场景 |
|------|---------|---------|
| quick | 仅 implementer | 简单修改 |
| standard | explorer + implementer | 常规开发任务 |
| full | explorer + implementer + reviewer | 复杂任务，需要代码审查 |

### 2.2 LLM 驱动的战略决策

战略决策通过 MCP `sampling/createMessage` 请求 supervisor（LLM）完成，替代了原来的规则引擎 `CrewDecisionPolicy`。

| 层级 | 职责 | 执行者 |
|------|------|--------|
| 战略层 | spawn Worker、accept、challenge | Supervisor（LLM，通过 sampling） |
| 战术层 | 轮询 Worker、自动验证、自动挑战 | Supervision Loop（Python） |

Supervisor 是可替换的：
- **Codex**: MCP Client 内置 sampling callback
- **Claude API**: 自定义 async callback 调用 Anthropic SDK
- **人工 Operator**: 终端交互，人工输入决策

### 2.3 对抗性验证循环

系统实现了一个完整的挑战-修复-验证循环：

1. Worker 完成任务后，监督者运行验证命令
2. 验证失败时，系统自动发出挑战（challenge）并要求修复（repair）
3. Worker 尝试修复后再次验证
4. 三次失败后升级给 supervisor 做战略决策

### 2.4 安全策略门控

- **命令策略**: 阻止 `rm -rf`、`git reset --hard`、`git clean -fd` 等破坏性命令
- **Shell 内联执行**: 阻止 `python -c`、`node -e`、`sh -c` 等内联执行
- **路径保护**: 保护 `.env`、`.git/`、`secrets/`、`*.pem`、`*.key` 等敏感路径
- **写入范围门控**: Worker 只能修改其分配的写入范围内的文件

### 2.5 Git Worktree 隔离

Implementer 类型的 Worker 获得独立的 Git worktree，分支命名规则为 `codex/{crew_id}-{worker_id}`，确保多个 Worker 的代码修改互不干扰。

### 2.6 黑板模式（Blackboard Pattern）

所有 Worker 通过共享黑板交换信息，条目类型包括：`fact`（事实）、`claim`（声明）、`question`（问题）、`risk`（风险）、`patch`（补丁）、`verification`（验证结果）、`review`（审查意见）、`decision`（决策）。

通过 MCP Server 的 Context Layer 进行过滤和摘要，防止上下文膨胀。

### 2.7 治理式学习系统

- 自动从重复失败中创建学习笔记和护栏候选
- 跟踪 Worker 质量评分
- 新技能和护栏需要人工审批后才能激活
- 安全扫描：检测 API 密钥、私钥、策略绕过模式

### 2.8 Worker 间消息传递

通过 `AgentMessageBus` 实现 Worker 间通信，消息类型：`handoff`（交接）、`question`（提问）、`answer`（回答）、`evidence`（证据）、`status`（状态）。消息通过 `<<<CODEX_MESSAGE ...>>>` 标记块从 Worker 输出中解析。

### 2.9 可视化控制台

内置 HTTP 服务器提供 Web UI，端点包括：
- `/` — 主页面
- `/api/state` — 整体状态
- `/api/sessions/{id}` — 会话详情
- `/api/runs/{id}` — 运行详情
- `/api/skills/{id}` — 技能详情

## 4. 架构设计

### 4.1 整体架构

```
MCP Client（Supervisor，可替换）
┌──────────────────────────────────┐
│  Codex / Claude API / 任意 agent │
│  接收 sampling 请求，返回决策     │
└──────────────┬───────────────────┘
               │ MCP protocol (stdio)
               │ sampling/createMessage (server→client)
               ▼
┌──────────────────────────────────────────────────────────┐
│  MCP Server (Python, FastMCP)                            │
│                                                          │
│  crew_run(crew_id, max_rounds, verification_commands)    │
│    │                                                     │
│    └─ 内部循环:                                          │
│        _wait_for_workers() 阻塞轮询                       │
│        → Worker 完成 → 自动验证                          │
│        → 验证通过 → sampling → "accept 吗?"              │
│        → 验证失败 < 3 → 自动挑战 → 继续                  │
│        → 验证失败 >= 3 → sampling → "你决定"              │
│        → max_rounds → 返回最终状态                       │
│                                                          │
│  crew_start / crew_stop / crew_status                    │
│  crew_accept / crew_challenge                            │
│  crew_blackboard / crew_events / crew_observe / ...      │
└──────────────┬──────────────────────────────────────────┘
               │
               ▼
┌──────────────────────────────────────────────────────────┐
│  CrewController / WorkerPool / Blackboard                │
└──────────────┬──────────────────────────────────────────┘
               │
        ┌──────┼──────┐
        ▼      ▼      ▼
   tmux Worker tmux Worker tmux Worker
```

### 4.2 控制流对比

| | 旧（Codex 驱动） | 新（Server 驱动） |
|--|------------------|------------------|
| 循环在哪 | Codex 反复调 tool | MCP Server 内部 |
| 轮询 | run_step 不阻塞，瞬间耗尽 | _wait_for_workers 阻塞轮询 |
| 决策 | tool 返回 → Codex 再调 | sampling 请求 → Supervisor 回复 |
| supervisor | 绑死 Codex | 可替换任意 MCP client |
| crew_run | Codex 调多次 | 调一次等最终结果 |
| 规则引擎 | CrewDecisionPolicy fallback | 已删除，完全由 LLM 决策 |

### 4.3 两代运行时架构

#### V3 — 基于 JSONL 文件的状态持久化

- 状态存储在 JSONL 文件中（append-only 流）：`workers.jsonl`、`events.jsonl`、`decisions.jsonl`、`messages.jsonl` 等
- 可变状态存储在 JSON 文件中：`crew.json`、`tasks.json`、`team_snapshot.json`
- 通过 `CrewRecorder` 类管理，支持乐观锁的状态转换

#### V4 — 事件溯源架构

- 所有状态变更为不可变事件，存储在事件存储中（SQLite 或 PostgreSQL）
- 通过投影（Projection）从事件流重建物化视图
- 支持幂等追加（idempotent append），使用内容摘要作为幂等键
- 事件类型包括：`crew.started`、`worker.registered`、`turn.delivered`、`turn.completed`、`scope.evaluated`、`review.verdict`、`challenge.issued` 等

### 4.4 关键设计模式

| 模式 | 应用位置 | 说明 |
|------|---------|------|
| MCP Server 驱动 | MCP Server | Server 内部运行完整监督循环 |
| Sampling 决策 | _ask_supervisor | 通过 MCP sampling 请求 LLM 做战略决策 |
| 可替换 Supervisor | sampling_fn 回调 | 任意 MCP client 可作为 supervisor |
| 战略/战术分层 | Supervision Loop | 战略决策由 LLM，战术执行由自动循环 |
| Context Layer | MCP Server | 摘要/过滤/压缩，防止上下文膨胀 |
| 事件溯源 | V4 EventStore | 不可变事件流 + 投影重建状态 |
| 契约驱动 | WorkerContract | Worker 通过契约定义能力与权限 |
| 黑板模式 | BlackboardEntry | 共享知识库，通过 Context Layer 消费 |
| 策略门控 | PolicyGate / WriteScopeGate | 安全策略强制执行 |
| 对抗性验证 | AdversarialEvaluator | 挑战-修复-验证循环 |
| 乐观锁 | transition_worker_status | CAS 语义的 Worker 状态转换 |

### 5.2 核心层 (`core/`)

#### `models.py` — 核心领域模型

定义 V1/V2 运行时的领域模型：

- `TaskRecord`: 任务记录，包含 goal、workspace_path、status 等字段
- `RunRecord`: 运行记录
- `EventRecord`: 事件记录
- `SessionRecord`: 会话记录，包含 task_id、agent_profile、workspace_mode、status 等
- `TurnRecord`: 轮次记录
- `ChallengeRecord`: 挑战记录
- `VerificationRecord`: 验证记录
- `LearningNote`: 学习笔记
- `SkillRecord`: 技能记录

枚举类型：
- `WorkspaceMode`: READONLY / SHARED / ISOLATED
- `TaskStatus`: pending / running / completed / failed
- `SessionStatus`: pending / running / completed / failed / needs_human
- `TurnPhase`: thinking / acting / done

#### `policy_gate.py` — 安全策略门控

`PolicyGate` 类实现命令和路径级别的安全检查：

```python
# 阻止的命令模式
BLOCKED_COMMANDS = ["rm -rf", "git reset --hard", "git clean -fd"]

# 阻止的 Shell 内联执行
BLOCKED_SHELL_INLINE = ["python -c", "node -e", "sh -c"]

# 保护的路径模式
PROTECTED_PATHS = [".env", ".git/", "secrets/", "*.pem", "*.key"]
```

### 5.3 适配器层 (`adapters/`)

#### `claude_cli.py` — Claude CLI 子进程适配器

构建 `claude --print ... --output-format json` 命令并解析结构化 JSON 输出。负责将任务描述转换为 Claude CLI 可执行的命令行参数。

### 5.4 桥接层 (`bridge/`)

#### `claude_bridge.py` — 长对话 Claude 桥接

管理长生命周期、可恢复的 Claude CLI 对话，使用 JSONL 进行轮次追踪。

主要方法：
- `start()`: 启动新对话

`CrewSupervisorLoop` 是核心编排类，包含 35 个方法。

**MCP 模式下的 `run()` 方法**（async）：

```python
async def run(
    self,
    *,
    crew_id: str,
    max_rounds: int,
    verification_commands: list[str],
    sampling_fn,  # async (messages, system_prompt, max_tokens) -> CreateMessageResult
) -> dict:
```

内部循环：
1. `_wait_for_workers(crew_id)` — 阻塞轮询，直到所有 Worker 完成
2. `_auto_verify(crew_id, verification_commands)` — 自动验证
3. 验证通过 → `_ask_supervisor()` → accept 或执行其他决策
4. 验证失败 >= 3 次 → `_ask_supervisor()` → spawn/challenge/accept
5. 验证失败 < 3 次 → `_auto_challenge()` — 自动挑战

**`_ask_supervisor()`** 通过 MCP sampling 请求 supervisor：
- 压缩当前 Crew 状态（`compress_crew_status`）

- 执行决策（`_execute_decision`）

适用于已启动的 Crew，执行完整的监督循环：explorer → implementer → reviewer，带验证和挑战。

## 7. MCP Server（已实现）

### 7.1 概述

MCP Server 基于 FastMCP 实现，通过 stdio 协议运行。Server 内部驱动完整监督循环，通过 MCP `sampling/createMessage` 请求 supervisor（LLM）做战略决策。

### 7.2 MCP Tools 清单（11 个）

**Crew 生命周期** (3 个):

| Tool | 说明 |
|------|------|
| `crew_start` | 启动 Crew，指定 repo、goal、roles |
| `crew_stop` | 停止 Crew |
| `crew_status` | 获取压缩状态（Context Layer） |

**战略决策** (2 个):

| Tool | 说明 |
|------|------|
| `crew_accept` | 接受结果，触发合并 |
| `crew_challenge` | 对 Worker 发出自定义挑战 |

**上下文获取** (5 个，带压缩):

| Tool | 说明 |
|------|------|
| `crew_blackboard` | 读取过滤后的黑板条目 |
| `crew_events` | 读取关键事件 |
| `crew_observe` | 观察 Worker 当前轮次输出 |
| `crew_changes` | 查看文件变更 |
| `crew_diff` | 查看特定文件 diff |

**战术执行** (1 个):

| Tool | 说明 |
|------|------|
| `crew_run` | 长时间运行，内部完整监督循环，通过 sampling 与 supervisor 交互 |

### 7.3 crew_run 详细实现

```python
@server.tool("crew_run")
async def crew_run(
    crew_id: str,
    ctx: Context,  # FastMCP 自动注入
    max_rounds: int = 3,
    verification_commands: list[str] | None = None,
) -> list[TextContent]:
    result = await supervision_loop.run(
        crew_id=crew_id,
        max_rounds=max_rounds,
        verification_commands=verification_commands or [],
        sampling_fn=lambda msgs, sys_prompt, max_tok: ctx.session.create_message(
            messages=msgs, max_tokens=max_tok, system_prompt=sys_prompt,
        ),
    )
    return [TextContent(type="text", text=json.dumps(result))]
```

### 7.4 MCP Sampling API

Server 端通过 `ctx.session.create_message()` 请求 supervisor：

```python
result = await ctx.session.create_message(
    messages=[types.SamplingMessage(role="user", content=types.TextContent(type="text", text=prompt))],
    max_tokens=500,
    system_prompt="你是 Crew supervisor，负责战略决策。",
)
reply = result.content.text  # "accept" / "spawn_worker(label='fixer', mission='...')" / "challenge(worker_id='w1', goal='...')"
```

Client 端必须提供 `sampling_callback`。如果 client 不支持 sampling，返回 `ErrorData(code=INVALID_REQUEST, message="Sampling not supported")`。

### 7.5 Supervisor 可替换性

MCP Server 不关心谁是 supervisor。`sampling_fn` 是一个 async 回调：

```python
# 标准 MCP sampling（通过 ctx.session）
sampling_fn = lambda msgs, sys, tok: ctx.session.create_message(...)

# 自定义 LLM
async def custom_sampling(messages, system_prompt, max_tokens):
    return await my_llm_client.generate(...)

# 人工 operator
async def human_sampling(messages, system_prompt, max_tokens):
    print(messages[0].content.text)
    reply = input("Your decision: ")
    return CreateMessageResult(role="assistant", content=TextContent(type="text", text=reply), model="human")
```

### 7.6 Context Layer

核心价值——MCP Server 不是简单转发，而是对数据做智能压缩：

- `crew_status`: 返回结构化摘要（非原始 dump），每个 Worker 只保留 id/role/status/summary
- `crew_blackboard`: 按 worker_id/type 过滤，默认 limit=10，超过 20 条自动摘要
- `crew_events`: 只返回关键事件（turn.completed/failed/challenge/review），跳过中间事件
- `crew_observe`: 只返回当前轮次文本（非全量 tmux 快照）
- 每个 tool 有 `max_tokens` 参数（默认 2000），超过自动截断

### 7.7 已删除的 Tools

| Tool | 原因 |
|------|------|
| `crew_decide` | 决策通过 sampling 自然发生 |
| `crew_spawn` | supervisor 在 sampling 回复中指定 spawn |
| `crew_verify` | 验证是 crew_run 内部自动的 |
| `crew_merge_plan` | 合并是 accept 的一部分 |

## 8. 数据流与工作流程

### 8.1 MCP 模式下的 Crew 运行流程

```
1. MCP Client 调用 crew_start(repo, goal, roles)
       │
2. MCP Server 创建 CrewController，启动 Worker
       │
3. MCP Client 调用 crew_run(crew_id, max_rounds, verification_commands)
       │
4. MCP Server 内部循环：
   ┌──────────────────────────────────────────────────┐
   │ a. _wait_for_workers() 阻塞轮询                   │
   │    → 所有 Worker idle/stopped/failed              │
   │                                                   │
   │ b. _auto_verify() 自动验证                         │
   │    → 运行 verification_commands                    │
   │                                                   │
   │ c. 验证通过?                                       │
   │    → _ask_supervisor() sampling 请求               │
   │    → supervisor 回复 "accept" → _do_accept()       │
   │    → supervisor 回复 "spawn_worker(...)" → 执行    │
   │    → supervisor 回复 "challenge(...)" → 执行       │
   │                                                   │
   │ d. 验证失败 >= 3 次?                               │
   │    → _ask_supervisor() sampling 请求               │
   │    → 执行 supervisor 决策                          │
   │                                                   │
   │ e. 验证失败 < 3 次?                                │
   │    → _auto_challenge() 自动挑战                    │
   │    → 回到 a                                        │
   └──────────────────────────────────────────────────┘
       │
5. 返回最终结果给 MCP Client
```

### 8.2 CLI 模式下的 Crew 运行流程

```
1. 用户指定目标 (goal)
       │
2. CLI 解析命令，创建 CrewController
       │
3. CrewController.start() 或 start_dynamic()
       │
4. 创建 Worker 契约 (WorkerContract)
       │
5. 为每个 Worker 准备工作空间
       │
6. 启动 tmux 会话，运行 Claude CLI
       │
7. Supervision Loop (supervise/supervise_dynamic):
   → 发送任务 → 等待完成 → 验证 → 挑战/修复 → 接受
       │
8. 合并 Worker 补丁到主分支
       │
9. 生成最终报告
```

### 8.3 V4 事件流

```
crew.started
    │
    ├── worker.registered (per worker)
    │       │
    │       ├── turn.delivered
    │       │       │
    │       │       ├── scope.evaluated
    │       │       ├── review.verdict
    │       │       ├── turn.completed / turn.failed / turn.timeout
    │       │       └── message.acknowledged
    │       │
    │       └── challenge.issued (if needed)
    │               │
    │               ├── repair.requested
    │               └── repair.completed
    │
    ├── readiness.evaluated
    ├── worker.patch.recorded
    ├── worker.result.recorded
    └── crew.ready_for_accept
```

### 8.4 Worker 生命周期

```
created → idle → busy → idle → ... → stopped
                    │
                    └─ (crash) → recovered → idle
```

状态转换使用乐观锁（CAS 语义）：
```python
transition_worker_status(worker_id, expected_status="busy", new_status="idle")
```

## 12. 已知缺点与不足

### 12.1 架构层面

1. **两代架构并存**: V3（JSONL 文件持久化）和 V4（事件溯源）同时存在，增加了代码复杂度和维护成本。

2. **缺乏运行时依赖管理**: `pyproject.toml` 声明仅依赖 `mcp`，但系统实际依赖 tmux、Claude CLI、Git 等外部工具，没有在包级别进行声明和检查。

3. **CLI 入口过于庞大**: `cli.py` 约 1150 行，包含所有命令的解析和执行逻辑，违反单一职责原则。

4. **缺少统一的错误处理框架**: 各模块的错误处理方式不一致。

### 12.2 可靠性层面

5. **tmux 依赖的脆弱性**: 系统高度依赖 tmux 窗格捕获来检测 Worker 完成状态，标记检测基于字符串匹配。

6. **崩溃恢复不完整**: 对于 Worker 在关键操作中途崩溃的情况，缺乏完善的回滚机制。

7. **事件存储 SQLite 的并发限制**: SQLite 在高并发写入场景下可能成为瓶颈。

### 12.3 功能层面

8. **缺乏 Worker 优先级调度**: 当前的 Worker 选择基于能力匹配和质量评分，缺乏优先级队列和抢占机制。

9. **学习系统的保守性**: 治理式学习要求人工审批才能激活新技能和护栏。

10. **合并冲突处理简单**: 仅检测文件所有权冲突，无法处理语义级别的合并冲突。

11. **缺乏资源限制**: 没有对 Worker 的 CPU、内存、磁盘使用进行限制。

### 12.4 可观测性层面

12. **日志体系不完善**: 缺乏结构化日志和日志级别控制。

13. **Web UI 功能有限**: 仅提供基本的状态查看功能，缺乏实时更新和交互操作。

14. **缺乏性能指标收集**: 没有对 Worker 执行时间、验证通过率等关键指标进行系统性收集。

### 12.5 安全层面

15. **策略门控可绕过**: `PolicyGate` 基于字符串匹配阻止危险命令，可通过编码、变量替换等方式绕过。

16. **Worker 隔离不彻底**: 所有 Worker 共享同一个文件系统和网络环境。

### 12.6 扩展性层面

17. **Protocol 接口版本管理缺失**: 缺乏版本管理和向后兼容策略。

18. **Agent 包系统原始**: 仅是 Markdown 文件片段，缺乏结构化的元数据和版本管理。
