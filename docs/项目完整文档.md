## 1. 项目概述

**codex-claude-orchestrator** 是一个本地化的多 Agent 编排系统，用于管理和协调多个 Claude Code 工作实例。系统以 tmux 会话为运行容器，通过 MCP Server 暴露编排能力，支持 LLM 驱动的战略决策。

### 核心定位

- **统一 Agent 架构**: Supervisor 不是框架原语——它只是另一个 Agent（Claude CLI in tmux），拥有 MCP Tools + 编排 Skill。所有 Agent（supervisor 和 worker）共享相同的调用类型
- **编排即 Skill**: 编排逻辑存在于提示词文件（`skills/orchestration-default.md`），而非 Python 代码。修改编排规则 = 编辑 Markdown
- **MCP Server = 纯基础设施**: 没有 `crew_run`、没有 sampling、没有监督循环。MCP Server 只提供 Tools：spawn、observe、blackboard、verify、accept、challenge
- **本地运行**: 所有组件在本地机器上运行，不依赖云服务
- **契约驱动**: Worker 通过 `WorkerContract` 定义能力、权限、工作空间策略和验收标准
- **事件溯源 (V4)**: 最新版本采用事件溯源架构，所有状态变更为不可变事件

### 入口点

```
orchestrator = "codex_claude_orchestrator.cli:main"
```

定义在 `pyproject.toml` 中，通过 `orchestrator` 命令行工具调用。

## 2. 核心功能

### 2.1 多 Agent 编排

系统可以同时管理多个 Claude Code 工作实例，每个实例运行在独立的 tmux 窗格中。支持三种预设角色组合：

| 模式 | 角色组成 | 适用场景 |
|------|---------|---------|
| quick | 仅 implementer | 简单修改 |
| standard | explorer + implementer | 常规开发任务 |
| full | explorer + implementer + reviewer | 复杂任务，需要代码审查 |

### 2.2 统一 Agent 架构

Supervisor 和 Worker 都是 Agent——都是 Claude CLI 进程，运行在 tmux 中。区别仅在于 Supervisor 拥有 MCP Tools + 编排 Skill。

| 角色 | 运行方式 | 拥有的能力 |
|------|---------|-----------|
| Supervisor | Claude CLI in tmux | MCP Tools（spawn/observe/accept/challenge）+ 编排 Skill |
| Worker | Claude CLI in tmux | 源码读写 + 黑板写入 + 完成标记 |

编排逻辑存在于 `skills/orchestration-default.md`，定义了：
- Worker 模板（targeted-code-editor、repo-context-scout 等）
- 编排循环（spawn → observe → verify → accept/challenge）
- 决策指南（何时 challenge、何时 spawn 新 worker）

修改编排规则只需编辑 Markdown 文件，无需改动 Python 代码。

### 2.3 对抗性验证循环

系统实现了一个完整的挑战-修复-验证循环：

1. Worker 完成任务后，监督者运行验证命令
2. 验证失败时，系统自动发出挑战（challenge）并要求修复（repair）
3. Worker 尝试修复后再次验证
4. 三次失败后升级给 supervisor 做战略决策

### 2.4 安全策略门控

- **命令策略**: 阻止 `rm -rf`、`git reset --hard`、`git clean -fd` 等破坏性命令
- **Shell 内联执行**: 阻止 `python -c`、`node -e`、`sh -c` 等内联执行
- **路径保护**: 保护 `.env`、`.git/`、`secrets/`、`*.pem`、`*.key` 等敏感路径
- **写入范围门控**: Worker 只能修改其分配的写入范围内的文件

### 2.5 Git Worktree 隔离

Implementer 类型的 Worker 获得独立的 Git worktree，分支命名规则为 `codex/{crew_id}-{worker_id}`，确保多个 Worker 的代码修改互不干扰。

### 2.6 黑板模式（Blackboard Pattern）

所有 Worker 通过共享黑板交换信息，条目类型包括：`fact`（事实）、`claim`（声明）、`question`（问题）、`risk`（风险）、`patch`（补丁）、`verification`（验证结果）、`review`（审查意见）、`decision`（决策）。

通过 MCP Server 的 Context Layer 进行过滤和摘要，防止上下文膨胀。

### 2.7 治理式学习系统

- 自动从重复失败中创建学习笔记和护栏候选
- 跟踪 Worker 质量评分
- 新技能和护栏需要人工审批后才能激活
- 安全扫描：检测 API 密钥、私钥、策略绕过模式

### 2.8 Worker 间消息传递

通过 `AgentMessageBus` 实现 Worker 间通信，消息类型：`handoff`（交接）、`question`（提问）、`answer`（回答）、`evidence`（证据）、`status`（状态）。消息通过 `<<<CODEX_MESSAGE ...>>>` 标记块从 Worker 输出中解析。

### 2.9 可视化控制台

内置 HTTP 服务器提供 Web UI，端点包括：
- `/` — 主页面
- `/api/state` — 整体状态
- `/api/sessions/{id}` — 会话详情
- `/api/runs/{id}` — 运行详情
- `/api/skills/{id}` — 技能详情

### 4.2 统一 Agent 模型

Supervisor 和 Worker 的本质相同——都是运行在 tmux 中的 Claude CLI 进程：

| 维度 | Supervisor | Worker |
|------|-----------|--------|
| 运行时 | Claude CLI in tmux | Claude CLI in tmux |
| 拥有的能力 | MCP Tools（12 个） | 源码读写 + 黑板写入 |
| 决策方式 | 读取编排 Skill，调用 Tools | 根据任务描述执行 |
| 契约 | 无（自由决策） | WorkerContract（约束能力与权限） |
| 工作空间 | 无（不直接修改代码） | worktree / readonly / shared |

编排逻辑不嵌入 MCP Server，而是作为 Skill 文件存在。Supervisor Agent 读取 Skill 后，自主决定如何使用 Tools 完成编排。

### 4.3 统一 CQRS 架构

#### 什么是 CQRS？

CQRS（Command Query Responsibility Segregation，命令查询职责分离）是一种架构模式，核心思想是：

- **写操作（Command）** 和 **读操作（Query）** 使用不同的路径和数据结构
- 写入端只负责记录"发生了什么事"（事件），不直接修改可变状态
- 读取端从写入的事件流中"重建"出当前状态

#### 为什么用 CQRS？

融合前的问题：系统有两套并行存储（V3 JSONL + V4 EventStore），同一个数据要写两遍，容易不一致。融合后：

```
融合前（双写，容易不一致）:
  CrewController → CrewRecorder (JSONL)     ← V3
  V4CrewRunner   → EventStore (SQLite/PG)   ← V4
  同一份数据写了两遍！

融合后（单一写入源）:
  CrewController → DomainEventEmitter → EventStore  ← 唯一写入
                         ↓
                   不可变事件流（append-only）
```

#### 具体怎么工作？

**第一步：发生了一件事**（比如启动 Crew）

```python
# CrewController.start() 被调用
# 它通过 DomainEventEmitter 写入一条事件：
emitter.emit_crew_started(crew_id="c1", goal="实现登录功能")
# → EventStore 里多了一条记录：
#   type="crew.started", crew_id="c1", payload={"goal": "实现登录功能"}
```

**第二步：想了解当前状态时，从事件流"回放"出来**

```python
# 用户执行 `orchestrator crew status --crew-id c1`
# 系统从 EventStore 读取 c1 的所有事件，逐条回放：
#   1. crew.started  → Crew 状态 = running, goal = "实现登录功能"
#   2. worker.spawned → workers 列表多了 w1
#   3. worker.claimed → w1 状态 = busy
#   4. verification.passed → 验证通过
#   → 最终输出当前完整状态
```

这就是"投影"（Projection）——从事件流重建出物化视图。

#### 写入路径详解

```
你写的代码（CrewController / WorkerPool / BlackboardStore）
    │
    ▼
DomainEventEmitter          ← 类型化封装，提供 emit_crew_started()、emit_worker_spawned() 等方法
    │
    ▼
EventStore.append()         ← 写入 SQLite 或 PostgreSQL，不可变、append-only
    │
    ▼
事件流（永远只增不删）       ← 这就是"事件溯源"——所有历史都在
```

关键特性：
- **不可变**：事件一旦写入就永不修改或删除
- **幂等**：同一个事件写两次不会产生副作用（通过幂等键 `{crew_id}/{event_type}/{entity_id}` 去重）
- **有序**：每个事件有 sequence 号，保证顺序

#### 读取路径详解

```
EventStore（事件流，存在 SQLite/PG 数据库里，人类看不懂原始事件）
    │
    ├─→ CrewStateProjection（内存投影）
    │       │  从事件流重建，存在 Python 内存里
    │       │  速度极快，但关机就没了
    │       ▼
    │   V4CrewRunner / CLI / MCP Server（程序内部消费）
    │
    └─→ CrewRecorder（JSONL 投影）
            │  从事件流生成，写到磁盘文件
            │  workers.jsonl、blackboard.jsonl ...
            ▼
        人类直接查看（cat、编辑器、grep）
```

#### 为什么需要两种投影？

它们解决不同的问题：

| | CrewStateProjection | CrewRecorder (JSONL) |
|---|---|---|
| **存在哪** | Python 内存 | 磁盘文件 |
| **速度** | 极快（内存读取） | 较慢（文件 I/O） |
| **关机后** | 消失 | 保留 |
| **谁来读** | 程序代码（V4CrewRunner、CLI、MCP） | 人类（终端 cat/grep）+ 旧代码 |
| **格式** | Python dict | JSONL 文本（每行一个 JSON） |
| **用途** | 程序内部高效查询 | 调试、日志审计、向后兼容 |

举个例子：

```bash
# 人类想看 Crew 状态 → 读 JSONL 文件（直观，每行一条记录）
$ cat .codex/crews/c1/workers.jsonl
{"worker_id":"w1","role":"implementer","status":"idle",...}
{"worker_id":"w2","role":"reviewer","status":"busy",...}

# 程序想查 Crew 状态 → 用内存投影（快，直接拿 dict）
state = CrewStateProjection.from_events(events)
state.to_read_crew_dict()["workers"]  # → [{"worker_id":"w1",...}, {"worker_id":"w2",...}]
```

**核心原因**：EventStore 里的原始事件（SQLite/PG 表）对人类不友好——你不会想直接去查数据库表。JSONL 文件是给人看的"翻译版"。内存投影是给程序用的"高性能版"。两者都从同一个事件流生成，数据一定一致。

#### 一个完整例子：Worker 验证失败后的挑战流程

```
1. controller.verify(crew_id="c1", command="pytest", worker_id="w1")
   → 运行 pytest，发现失败
   → DomainEventEmitter.emit_verification_failed(c1, w1, "pytest", result)
   → EventStore 写入: type="verification.failed", ...

2. controller.challenge(c1, summary="修复测试失败", worker_id="w1")
   → DomainEventEmitter.emit_challenge_issued(c1, w1, "修复测试失败")
   → DomainEventEmitter.emit_repair_requested(c1, w1, "修复测试失败")
   → EventStore 写入: type="challenge.issued", ...
   → EventStore 写入: type="repair.requested", ...

3. 用户执行 `orchestrator crew events --crew-id c1`
   → 从 EventStore 读取所有事件
   → 输出：
     [1] crew.started        goal="实现登录功能"
     [2] worker.spawned      worker_id="w1"
     [3] verification.failed command="pytest"
     [4] challenge.issued    summary="修复测试失败"
     [5] repair.requested    instruction="修复测试失败"
```

每一步都只写 EventStore 一次，读取时从事件流重建——这就是 CQRS 的核心。

#### 事件类型完整清单

| 类别 | 事件类型 | 发出方 |
|------|---------|--------|
| Crew 生命周期 | `crew.started` / `crew.updated` / `crew.stopped` / `crew.finalized` | CrewController |
| Worker 生命周期 | `worker.spawned` / `worker.contract.recorded` / `worker.claimed` / `worker.released` / `worker.stopped` | CrewController / WorkerPool |
| 黑板 | `blackboard.entry` | CrewController |
| 决策 | `decision.recorded` | CrewController |
| 任务 | `task.created` | CrewController |
| 工件 | `artifact.written` | CrewController |
| 验证 | `verification.passed` / `verification.failed` | CrewController |
| 挑战与修复 | `challenge.issued` / `repair.requested` | CrewController |
| 审查 | `review.completed` | CrewController |
| 陷阱 | `pitfall.recorded` | CrewController |
| 轮次 | `turn.delivered` / `turn.completed` / `turn.failed` / `turn.timeout` | V4Supervisor |
| 验收 | `crew.ready_for_accept` / `crew.accepted` | V4WorkflowEngine |

### 4.4 关键设计模式

| 模式 | 应用位置 | 说明 |
|------|---------|------|
| 统一 Agent | 全局 | Supervisor 和 Worker 都是 Agent，共享运行时 |

### 6.1 事件原语 (`events.py`)

#### EventStore Protocol (`event_store_protocol.py`)

```python
class EventStore(Protocol):
```

### 6.3 DomainEventEmitter (`domain_events.py`)

封装 `EventStore.append()`，提供类型化的 emit 方法和一致的幂等键约定。是 CrewController、WorkerPool、BlackboardStore 等写入 EventStore 的唯一接口。

```python
class DomainEventEmitter:
    def __init__(self, events: EventStore): ...

    # Crew 生命周期
    def emit_crew_started(self, crew_id, goal, ...) -> AgentEvent
    def emit_crew_updated(self, crew_id, updates) -> AgentEvent
    def emit_crew_stopped(self, crew_id, reason) -> AgentEvent
    def emit_crew_finalized(self, crew_id, status, summary) -> AgentEvent

    # Worker 生命周期
    def emit_worker_spawned(self, crew_id, worker_id, ...) -> AgentEvent
    def emit_worker_contract_recorded(self, crew_id, contract_id, ...) -> AgentEvent
    def emit_worker_claimed(self, crew_id, worker_id) -> AgentEvent
    def emit_worker_released(self, crew_id, worker_id) -> AgentEvent
    def emit_worker_stopped(self, crew_id, worker_id) -> AgentEvent

    # 黑板 / 决策 / 任务 / 工件
    def emit_blackboard_entry(self, crew_id, entry_id, ...) -> AgentEvent
    def emit_decision_recorded(self, crew_id, action_id, ...) -> AgentEvent
    def emit_task_created(self, crew_id, task_id, ...) -> AgentEvent
    def emit_artifact_written(self, crew_id, artifact_name, ...) -> AgentEvent

    # 验证 / 挑战 / 修复 / 审查
    def emit_verification_passed(self, crew_id, worker_id, command, ...) -> AgentEvent
    def emit_verification_failed(self, crew_id, worker_id, command, ...) -> AgentEvent
    def emit_challenge_issued(self, crew_id, worker_id, summary, ...) -> AgentEvent
    def emit_repair_requested(self, crew_id, worker_id, instruction, ...) -> AgentEvent
    def emit_review_completed(self, crew_id, worker_id, verdict_status, ...) -> AgentEvent

    # 陷阱
```

## 7. MCP Server（已实现）

### 7.1 概述

MCP Server 基于 FastMCP 实现，通过 stdio 协议运行。Server 是纯基础设施——只提供 Tools，不含循环、不含 sampling、不含决策逻辑。编排决策由 Supervisor Agent 通过 Skill 自主完成。

### 7.2 MCP Tools 清单（16 个）

**非阻塞编排** (3 个，Async Sub-agent 模式):

| Tool | 签名 | 说明 |
|------|------|------|
| `crew_run` | `(repo, goal, crew_id?, verification_commands?, max_rounds?)` | 非阻塞启动后台编排任务，返回 job_id + background_agent_prompt |
| `crew_job_status` | `(job_id)` | Delta 轮询：只返回变化（phase/round），未变化返回 "unchanged" |
| `crew_cancel` | `(job_id)` | 取消运行中的任务 |

**Crew 生命周期** (6 个):

| Tool | 签名 | 说明 |
|------|------|------|
| `crew_start` | `(repo, goal, roles)` | 启动 Crew |
| `crew_stop` | `(repo, crew_id)` | 停止整个 Crew |
| `crew_status` | `(repo, crew_id)` | 获取压缩状态（Context Layer） |
| `crew_spawn` | `(repo, crew_id, label, mission?)` | 生成 Worker（支持模板） |
| `crew_stop_worker` | `(repo, crew_id, worker_id)` | 停止指定 Worker |
| `crew_verify` | `(crew_id, command, worker_id?)` | 运行验证命令 |

**战略决策** (2 个):

| Tool | 签名 | 说明 |
|------|------|------|
| `crew_accept` | `(crew_id, summary?)` | 接受结果，触发合并 |
| `crew_challenge` | `(crew_id, summary, task_id?)` | 对 Worker 发出挑战 |

**上下文获取** (5 个，带压缩):

| Tool | 签名 | 说明 |
|------|------|------|
| `crew_blackboard` | `(crew_id, worker_id?, entry_type?)` | 读取过滤后的黑板条目 |
| `crew_events` | `(repo, crew_id)` | 读取关键事件 |
| `crew_observe` | `(repo, crew_id, worker_id)` | 观察 Worker 当前输出（结构化报告：status/summary/changed_files，非原始 tmux dump） |
| `crew_changes` | `(crew_id)` | 查看文件变更 |
| `crew_diff` | `(crew_id, file?)` | 查看特定文件 diff |

### 7.3 Worker 模板

`crew_spawn` 支持预定义模板，通过 `label` 参数快速创建常见 Worker：

| Label | Authority | Workspace | 适用场景 |
|-------|-----------|-----------|---------|
| `targeted-code-editor` | source_write | worktree | 实现代码修改 |

### 7.4 crew_spawn 实现

```python
WORKER_TEMPLATES: dict[str, WorkerContract] = {
    "targeted-code-editor": WorkerContract(
        contract_id="template-targeted-code-editor",
        label="targeted-code-editor",
        mission="Implement the requested changes in the source code.",
        required_capabilities=["inspect_code", "edit_source"],
        authority_level=AuthorityLevel.SOURCE_WRITE,
        workspace_policy=WorkspacePolicy.WORKTREE,
    ),
    # ... 其他模板
}

@server.tool("crew_spawn")
async def crew_spawn(repo: str, crew_id: str, label: str, mission: str = "") -> list[TextContent]:
    template = WORKER_TEMPLATES.get(label)
    if template:
        contract = replace(template, mission=mission or template.mission)
    else:
        contract = WorkerContract(
            contract_id=f"contract-{label}", label=label,
            mission=mission, required_capabilities=["inspect_code", "edit_source"],
            authority_level=AuthorityLevel.SOURCE_WRITE,
            workspace_policy=WorkspacePolicy.WORKTREE,
        )
    result = controller.ensure_worker(repo_root=Path(repo), crew_id=crew_id, contract=contract)
    return [TextContent(type="text", text=json.dumps(result, ensure_ascii=False))]
```

### 7.5 Context Layer

核心价值——MCP Server 不是简单转发，而是对数据做智能压缩：

- `crew_status`: 返回结构化摘要（非原始 dump），每个 Worker 只保留 id/role/status/summary
- `crew_blackboard`: 按 worker_id/type 过滤，默认 limit=10，超过 20 条自动摘要
- `crew_events`: 只返回关键事件（turn.completed/failed/challenge/review），跳过中间事件
- `crew_observe`: 返回结构化报告（status/summary/changed_files/risks），优先从 V4 outbox 读取，~300 token vs 原始 5000+ token
- 每个 tool 有 `max_tokens` 参数（默认 2000），超过自动截断

### 7.6 非阻塞 crew_run 架构

`crew_run` 是非阻塞的 MCP Tool，编排循环在 MCP Server 后台线程中自动运行。Claude Code 通过 Async Sub-agent 模式隔离轮询上下文。

**核心组件**：

| 组件 | 文件 | 职责 |
|------|------|------|
| `JobManager` | `mcp_server/job_manager.py` | Job 生命周期管理、Delta 状态追踪、TTL 清理 |
| `crew_run` | `mcp_server/tools/crew_run.py` | 非阻塞启动、返回 job_id + background_agent_prompt |
| `crew_job_status` | 同上 | Delta 轮询：只推变化，自适应间隔 |
| `crew_cancel` | 同上 | 通过 threading.Event 通知后台线程 |

**cancel 传播链**（通过参数传递，非共享状态）：
```
crew_cancel(job_id)
  → Job.cancel_event.set()
    → V4CrewRunner.supervise(cancel_event=...)
      → V4Supervisor.run_worker_turn(cancel_event=...)
        → ClaudeCodeTmuxAdapter.watch_turn(turn, cancel_event=...)
          → cancel.wait(delay) 返回 True → yield RuntimeEvent("runtime.cancelled")
```

**关键设计**：cancel_event 通过参数链传递（非 `adapter.set_cancel_event()` 共享状态），避免并发 job 互相覆盖。

**自适应轮询间隔**：

| 已用时间 | 轮询间隔 |
|----------|----------|
| < 5s     | 5s       |
| < 15s    | 10s      |
| < 35s    | 20s      |
| < 75s    | 40s      |
| ≥ 75s    | 60s      |

**Job 清理**：完成的 Job 在 1 小时后自动驱逐（TTL 机制）。

## 8. 数据流与工作流程

### 8.1 两种运行模式

系统支持两种运行模式，核心区别在于**谁来做编排决策**：

| | MCP 模式（统一 Agent 模式） | CLI 模式 |
|---|---|---|
| **谁做决策** | Supervisor Agent（Claude LLM） | Python 代码（CrewSupervisorLoop） |
| **决策方式** | LLM 读取编排 Skill，自主推理下一步 | 硬编码的循环逻辑（发任务→等完成→验证→挑战/接受） |
| **启动方式** | Supervisor Agent 通过 MCP 调用 `crew_spawn` 等 Tool | 用户执行 `orchestrator crew run --crew-id c1` |
| **编排逻辑在哪** | `skills/orchestration-default.md`（Markdown） | `crew/supervisor_loop.py`（Python） |
| **灵活性** | 高——LLM 可以根据上下文灵活决策 | 低——按固定流程执行 |
| **适合场景** | 复杂任务，需要智能判断 | 简单任务，流程固定 |

**共同点**：两种模式最终都调用同一个 `CrewController`（`start()`、`verify()`、`challenge()`、`accept()` 等方法），底层实现完全一样。区别只在于"谁来决定调哪个方法、什么时候调"。

#### MCP 模式流程（Supervisor Agent 做决策）

```
1. 启动 Supervisor Agent（Claude CLI in tmux，加载编排 Skill）
       │
2. Supervisor 通过 MCP Tools 执行编排：
   ┌──────────────────────────────────────────────────┐
   │ a. crew_spawn(label="targeted-code-editor")      │
   │    → 启动 Worker（Claude CLI in tmux）             │
   │                                                   │
   │ b. crew_observe(crew_id, worker_id)               │
   │    → 读取 Worker 输出                              │
   │                                                   │
   │ c. Worker 完成（检测到完成标记）                    │
   │    → crew_verify(crew_id, "pytest")               │
   │                                                   │
   │ d. 验证通过?                                       │
   │    → crew_accept(crew_id, summary="验证通过")      │
   │    → 触发合并                                      │
   │                                                   │
   │ e. 验证失败?                                       │
   │    → crew_challenge(crew_id, summary="修复 X")    │
   │    → Worker 修复 → 回到 c                         │
   │                                                   │
   │ f. 多次失败?                                       │
   │    → crew_spawn(label="verification-failure-..")  │
   │    → 或报告给人工                                  │
   └──────────────────────────────────────────────────┘
       │
3. 合并完成，Crew 结束
```

关键：循环不在 Python 代码里，而在 Supervisor Agent 的推理过程中。Supervisor 读取编排 Skill（Markdown），根据当前状态自主决定下一步调用哪个 Tool。

#### CLI 模式流程（Python 代码做决策）

```
1. 用户指定目标 (goal)
       │
2. CLI 解析命令，创建 CrewController
       │
3. CrewController.start() 或 start_dynamic()
       │
4. 创建 Worker 契约 (WorkerContract)
       │
5. 为每个 Worker 准备工作空间
       │
6. 启动 tmux 会话，运行 Claude CLI
       │
7. Supervision Loop (supervise/supervise_dynamic):
   → 发送任务 → 等待完成 → 验证 → 挑战/修复 → 接受
       │
8. 合并 Worker 补丁到主分支
       │
9. 生成最终报告
```

关键：编排循环是 Python 代码（`CrewSupervisorLoop`），按固定流程执行，不依赖 LLM 推理。

### 8.3 V4 事件流

```
crew.started
    │
    ├── worker.spawned (per worker)
    │   ├── worker.contract.recorded
    │   ├── worker.claimed / worker.released (per task)
    │   │
    │   ├── turn.delivered
    │   │       │
```

## 12. 已知缺点与不足

### 12.1 架构层面

1. **~~两代架构并存~~** (已解决): V3/V4 已融合为统一 CQRS 架构——EventStore 为唯一写入源，CrewRecorder 降级为只读投影。

2. **缺乏运行时依赖管理**: `pyproject.toml` 声明仅依赖 `mcp`，但系统实际依赖 tmux、Claude CLI、Git 等外部工具，没有在包级别进行声明和检查。

3. **CLI 入口过于庞大**: `cli.py` 约 1150 行，包含所有命令的解析和执行逻辑，违反单一职责原则。

4. **缺少统一的错误处理框架**: 各模块的错误处理方式不一致。

### 12.2 可靠性层面

5. **tmux 依赖的脆弱性**: 系统高度依赖 tmux 窗格捕获来检测 Worker 完成状态，标记检测基于字符串匹配。

6. **崩溃恢复不完整**: 对于 Worker 在关键操作中途崩溃的情况，缺乏完善的回滚机制。

7. **事件存储 SQLite 的并发限制**: SQLite 在高并发写入场景下可能成为瓶颈。

### 12.3 功能层面

8. **缺乏 Worker 优先级调度**: 当前的 Worker 选择基于能力匹配和质量评分，缺乏优先级队列和抢占机制。

9. **学习系统的保守性**: 治理式学习要求人工审批才能激活新技能和护栏。

10. **合并冲突处理简单**: 仅检测文件所有权冲突，无法处理语义级别的合并冲突。

11. **缺乏资源限制**: 没有对 Worker 的 CPU、内存、磁盘使用进行限制。

### 12.4 可观测性层面

12. **日志体系不完善**: 缺乏结构化日志和日志级别控制。

13. **Web UI 功能有限**: 仅提供基本的状态查看功能，缺乏实时更新和交互操作。

14. **缺乏性能指标收集**: 没有对 Worker 执行时间、验证通过率等关键指标进行系统性收集。

### 12.5 安全层面

15. **策略门控可绕过**: `PolicyGate` 基于字符串匹配阻止危险命令，可通过编码、变量替换等方式绕过。

16. **Worker 隔离不彻底**: 所有 Worker 共享同一个文件系统和网络环境。

### 12.6 扩展性层面

17. **Protocol 接口版本管理缺失**: 缺乏版本管理和向后兼容策略。

18. **Agent 包系统原始**: 仅是 Markdown 文件片段，缺乏结构化的元数据和版本管理。
